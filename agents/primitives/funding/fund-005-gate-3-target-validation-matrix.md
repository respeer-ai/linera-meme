# FUND-005 Gate 3 Target Validation Matrix

Type: Primitive
Audience: Coding assistants
Authority: High

## Purpose

Define the target validation matrix for `FUND-005` Gate 3. This matrix maps
the accepted Gate 1C protocol paths and the accepted Gate 2 current audit facts
to the target checks, state transitions, failure handling, projection truth, and
test obligations that must be satisfied before protocol mutation work begins.

## Rules

- Gate 3 is a target validation artifact, not an implementation patch.
- Every row must cite one accepted Gate 1C path.
- Target behavior must be stated as a single deterministic rule per row.
- Split rows when different entries, token legs, callbacks, receipts, failures,
  bounces, claim effects, projections, and tests require different validation.
- Do not use `webui/` as current frontend evidence.
- Use `webui-v2/` for current frontend and product-entry target work.
- Do not use generated GraphQL output under
  `webui-v2/src/__generated__/graphql/` as an audited source file.
- Do not touch Quasar framework files for Gate 3:
  `webui-v2/quasar.config.ts`,
  `webui-v2/src/boot/axios.ts`,
  `webui-v2/src/boot/i18n.ts`,
  `webui-v2/src/boot/notify-defaults.ts`,
  `webui-v2/src/boot/pinia.ts`.
- Runtime rows describe contract capabilities. Handler rows describe workflow
  ownership and business state transitions.
- `Claim` is the only user-facing funds-exit operation for pool-held value.
- Meme `Redeem` remains meme-local balance exit and does not credit pool claim
  balances.
- Pending forever is valid only when the state is safe, observable, and not
  finalized.
- Close-chain eligibility is target behavior and must not be treated as current
  runtime capability until `ContractRuntimeContext` exposes close execution.

## Gate 3 Target State Definitions

`CreateMemeIntent` is a target workflow state required by this matrix. Gate 3
defines its minimum validation shape so the CreateMeme rows are not referencing
an undefined state. Gate 4 must expand these states into the handler-level
transition table before any implementation change.

`CreateMemeIntent` is persisted by the proxy application on the proxy creator
chain. The proxy application allocates `intent_id` from its own monotonic
intent sequence before sending `ProxyMessage::CreateMeme`.

Required `CreateMemeIntent` fields:

- `intent_id`
- `creator_account`
- `signer_owner`
- `proxy_operation_chain`
- `proxy_creator_chain`
- `expected_meme_chain`
- `expected_meme_application`
- `expected_token`
- `initial_liquidity`
- `virtual_initial_liquidity`
- `status`
- `created_at_event`
- `last_update_event`
- `failure_reason`

`CreateMemeIntent` optional fields at creation:

- `expected_meme_chain`: `None` until `ProxyMessage::CreateMeme` opens the meme chain
- `expected_meme_application`: `None` until `ProxyMessage::CreateMemeExt` creates the meme application
- `expected_token`: `None` until `ProxyMessage::MemeCreated` carries the token application id

`CreateMemeIntent` required fields at creation:

- `intent_id`
- `creator_account`
- `signer_owner`
- `proxy_operation_chain`
- `proxy_creator_chain`
- `initial_liquidity`
- `virtual_initial_liquidity`
- `status`
- `created_at_event`
- `last_update_event`

Allowed `CreateMemeIntent.status` values:

- `pending`
- `creator_message_sent`
- `meme_chain_opened`
- `meme_application_created`
- `active`
- `failed`
- `stalled`
- `cleaned_up`

`MemeInitializeLiquidityRoute` is the target workflow state for meme-native pool
initialization. It is persisted by the swap application on the swap creator
chain. The meme application does not store an initialize-liquidity workflow
intent and does not track pool-shell, pool-created, activation, update, and pool
active states. Meme creation completes when the meme application state, token
metadata, mining state, balances, and allowance facts are committed. Direct pool
initialization is a protocol startup path driven by swap and pool handlers.

The meme-side `MemeMessage::LiquidityFunded` path carries the initialization
facts required for swap to create one route entry. Observability and product
status are derived by projection from meme creation facts, decoded protocol
messages, swap-side route state, and pool-side facts. Linera rejects same-stack
cross-application reentry when a call stack tries `App A -> App B -> App A`;
therefore `MemeOperation::InitializeLiquidity` must only queue
`MemeMessage::InitializeLiquidity` and must never synchronously call back into
swap. Funding success and failure receipts are emitted only later from
`MemeMessage::InitializeLiquidity` on the meme creator chain.

Required `MemeInitializeLiquidityRoute` fields:

- `route_id`
- `creator_account`
- `meme_application`
- `meme_creator_chain`
- `swap_application`
- `swap_creator_chain`
- `token0_identity`
- `token1_identity`
- `adjusted_fungible_amount`
- `native_amount`
- `virtual_initial_liquidity`
- `status`
- `created_at_event`
- `last_update_event`
- `failure_reason`

`MemeInitializeLiquidityRoute` optional fields at creation:

- `pool_application`: `None` until `SwapMessage::PoolCreated` carries the pool application account
- `funding_correlation_id`: `None` until `SwapMessage::PoolCreated` is accepted and the meme funding segment is started

Allowed `MemeInitializeLiquidityRoute.status` values:

- `accepted`
- `pool_shell_sent`
- `meme_funding_sent`
- `pool_created`
- `update_sent`
- `active`
- `failed`
- `stalled_funding`
- `stalled_update`
- `cleaned_up`

`route_id` is allocated by the swap application from swap creator-chain
persistent state when it accepts `SwapMessage::InitializeLiquidity`.
`funding_correlation_id` is an opaque value allocated by the swap application
when the swap creator chain accepts `SwapMessage::PoolCreated` and starts the
meme funding segment of the route. Meme handlers must not derive business
meaning from it; they only carry it through the funding operation, funding
message, and funding receipt.
Each meme application may create at most one `MemeInitializeLiquidityRoute`;
`meme_application` is the uniqueness key for initialization. After that route is
accepted, later liquidity entry is normal user add liquidity. The route stores
source-binding fields for validation: `meme_creator_chain`, `token0_identity`,
`creator_account`, `adjusted_fungible_amount`, `native_amount`,
`virtual_initial_liquidity`, and the bound `pool_application` once pool-shell
creation is accepted. The route stores one `native_amount` plus
`virtual_initial_liquidity`; implementation must expose helper APIs that
distinguish real native custody amount from virtual native reference amount
before reserve, TVL, claim, and payable-balance code can read the value.

Funding recovery for CreateMeme, meme initialization, pool creation, add
liquidity, swap, remove liquidity, and claim payout failures must credit
`claim_balances[token_identity][owner_account] += amount` when user-owned value
has entered protocol custody. Each matrix row that credits claim balances must
state the exact `token_identity`, `owner_account`, and `amount` source.
